IOC

==面试：讲讲Spring IOC？（1）是什么（2）怎么用（3）原理和源码==

==回答思路：是什么 + 怎么用（XML 注解） + 解释两种容器（BeanFactory ApplicationContext） + 底层原理==

IOC（控制反转/依赖注入）并不是我们主动获取依赖的对象，而是让容器帮你实例化对象并注入。

被注入对象和被依赖对象由 IoC Server Provider 统一管理，被注入对象需要什么，直接和 Ioc Server Provider 说一声，他就会把对应的被依赖对象注入进去。

IOC Service Provider 的职责

  • 业务对象的构建管理：

    业务对象无需关心所依赖对象如何取得，IOC Service Provider 将对象的构建逻辑从客户端剥离封装出来。

  • 业务对象间的依赖绑定：

    IOC Service Provider 通过结合之前构建和管理的所有业务对象，以及各个业务对象间可以识别的依赖关系，将这些对象所依赖的对象注入绑定，从而保证业务对象在使用的时候，处于就绪状态。

==3）注解方式（重要）==

Spring注入依赖的4个注解：

  • @Value 注入int、float、String等 基本数据类型，只能标注在成员变量、setter 方法上。
  • @Autowired 按 类型 自动装配，可标注在成员变量（官方不推荐）、构造方法、setter方法上。
  • @Qualifier 按 名称 自动装配，需要和 @Autowired 搭配使用，解决当有多个相同类型的 Bean 时，Spring 无法确定注入哪个 Bean 的问题。只能标注在成员变量（官方不推荐）、setter方法上。
  • @Resource 按 名称 或 类型 自动装配，需要第三方包 javax.annotation.jar 的支持，只能标注在成员变量、setter方法上。

以上3个注解用于自动装配其它bean的实例，尽量标注在setter方法上。

==@Autowired==
  • ==@Autowired 是Spring的核心注解之一，它可以用于自动装配Bean。当Spring容器发现一个需要被注入的Bean时，它会根据类型进行自动装配。==
  • 如果有多个候选Bean与被注入的类型匹配，Spring会尝试根据类型匹配选择正确的Bean，但如果找不到唯一匹配的Bean，会抛出异常。
==@Qualifier==
  • ==@Qualifier 通常与 @Autowired 配合使用，用于指定具体要注入的Bean的名称。通过指定具体的Bean名称，解决了====@Autowired 可能存在的多个匹配Bean的问题。==
  • 使用 @Qualifier 时，需要提供相应Bean的名称。
@Resource
  • @Resource 是Java EE提供的注解，也可以用于注入Bean。它可以按照名称进行注入，也支持按照类型进行注入。
  • 与 @Autowired 不同，@Resource 没有 required 属性，即默认按照名称进行注入，如果找不到对应的Bean则会抛出异常。
@Inject
  • @Inject 是Java标准的依赖注入注解，与 @Autowired 功能类似，也可以用于注入Bean。它是JSR-330的一部分，与 @Autowired 相比，使用 @Inject 可以使代码更加标准化。
  • @Inject 与 @Autowired 的主要区别在于来源，@Inject 是JSR-330定义的标准，而 @Autowired 是Spring提供的特定实现。
@Value
  • @Value 注解可以用来注入一个具体的值，例如字符串、数字等。这个值可以来自于配置文件中的属性值。
  • @Value 注解可以用于字段、构造方法或者Setter方法上，用来指定要注入的值。

这些注解各有特点，选择使用哪一个取决于具体的需求以及个人偏好。通常情况下，推荐使用 @Autowired 进行自动装配，配合 @Qualifier 来解决多个候选Bean的问题。

BeanFactory & ApplicationContext

==面试：讲讲IOC两种容器？==

Spring 提供了两种容器类型：（都负责 Bean 的生命周期和依赖注入）

  • BeanFactory

    基础类型 IOC 容器，负责创建和管理 Bean 的生命周期。默认采用懒加载（lazy-load），即只有访问容器中的对象时，才对对象执行初始化和依赖注入操作。

    因此，容器启动初期，对于资源有限，功能要求不严格的场景，BeanFactory 是合适的 IOC 容器选择。

    BeanFactory 可以理解成 HashMap，键是bean name，值是bean object，一般只有put、get两个功能

  • ApplicationContext（企业常用）

    在 BeanFactory 的基础上构建，除了有 BeanFactory 的功能，还有事件发布、国际化信息支持等功能。ApplicationContext 采用即时加载（Eager-load），在容器启动后，默认全部初始化并绑定完成。

    因此，需要更多系统资源，对于资源充足，功能要求更多的场景，ApplicationContext 是合适的 IOC 容器选择。

特性 BeanFactory ApplicationContext
加载方式 按需加载（懒加载） 即时加载（加载时初始化所有 Bean）
功能 基本功能 高级功能：事件、AOP、国际化等
性能 更高效，适合简单应用 功能强大，适合复杂应用
容器实现类 XmlBeanFactory（已废弃）和 DefaultListableBeanFactory ClassPathXmlApplicationContext、AnnotationConfigApplicationContext等
常见应用 嵌入式应用、资源有限的环境 企业级应用、Web 应用、复杂业务需求

1.4.1 统一资源加载策略

java.net.URL的问题是，资源查找后返回的形式没有一个统一的抽象，这样客户端不能统一处理。所以，Spring 提出了基于 org.springframework.core.io.Resource 和 org.springframework.core.io.ResourceLoader接口的资源抽象和加载策略。

1）Spring 的 Resource

Spring框架使用 org.springframework.core.io.Resource 接口作为所有资源的抽象和访问接口：

Beanfactory beanFactory = new XmlBeanFactory(new ClassPathResource("..."));

ClassPathResource 就是 Resource 的特定实现，代表位于 ClassPath 的资源。

2）Spring 的 ResourceLoader，“更广义的URL”

有了资源，怎么查找、定位？ResourceLoader来做！他是查找定位资源的抽象，具体策略由实现类给出。

3）ApplicationContext 和 ResourceLoader

ApplicationContext 继承了 ResourcePatternResolver，当 然就间接实现了 ResourceLoader 接口。所以，任何的 ApplicationContext 实现都可以看作是一个 ResourceLoader 甚至 ResourcePatternResolver 。而这就是 ApplicationContext 支持 Spring 内统一资源加载策略的真相。

AOP

==面试：讲一下AOP==

==思路：是什么 + 实现原理（动态代理）+ 底层原理==

AOP（Aspect Oriented Programming），即面向切面编程。AOP 允许你在不修改源代码的情况下，将新的功能添加到现有的业务逻辑中，这些新增的功能通常是一些与核心业务逻辑无关的公共任务，例如日志、事务管理、安全、性能监控等。

面向切面编程，是一种理念，要用语言实现，即 AOL。AOL 不一定与系统语言相同，比如 AspectJ 是拓展自 Java 的 AOL。（还有 AspectC、AspectC++、AspectPHP …）。AOL 要织入（Weave）到寄生语言中。

静态 AOP

第一代 AOP（AspectJ）的特点是响应的横切点以 Aspect 的形式实现后会通过特定编译器织入系统的静态类，没有性能损失，但灵活性不足。

动态 AOP

第二代 AOP 大多通过 Java 的动态特性实现 AspectJ 的植入系统过程，比如 JBoss AOP、Spring AOP 等。（第二代 AOL 大都 java 实现，Aspect 和 Class 都一样，易集成）

AOP 的织入过程是在系统开始后进行，而不是预编译到系统中。

2.1 Java 平台的 AOP 实现机制

在 Java 平台上可以使用多种方式实现 AOP。下面提到的几种方式是最经常使用的。

2.1.1 动态代理

JDK1.3 之后引入了动态代理 (Dynamic Proxy) 机制可以在运行期间，为相应的接口 (Interface) 动态生成对应的代理对象。所以，我们可以将横切逻辑封装到动态代理的 InvocationHandler 中，然后在系统运行期间，根据横切逻辑需要织入的模块位置，将横切逻辑织入到相应的代理类中。以动态代理类为载体的横切逻辑，现在当然就可以与系统其他实现模块一起工作了。

这种方式实现的唯一缺点或者说优点就是，所有需要织入横切逻辑的模块类都得实现相应的接口，因为动态代理机制只针对接口有效。当然，毕竟动态代理是在运行期间使用反射，相对于编译后的静态类的执行，性能上可能稍逊一些。

Spring AOP 默认情况下采用这种机制实现 AOP 功能。

2.1.2 动态字节码增强

在 Java 中，动态字节码增强技术是指在程序运行期间动态地修改Java类的字节码，以实现对类的功能增强或修改。这种技术通常用于实现 AOP（面向切面编程）、方法拦截、性能监控等需求。

我们可以为需要织入横切逻辑的模块类在运行期间，通过动态字节码增强技术为这些系统模块类生成相应的子类，而将横切逻辑加到这些子类中，让应用程序在执行期间使用的是这些动态生成的子类，从而达到将横切逻辑织入系统的目的。

使用动态字节码增强技术，即使模块类没有实现相应的接口，依然可以对其进行扩展，而不用像动态代理那样受限于接口。不过，这种实现机制依然存在不足，如果需要扩展的类以及类中的实例方法等声明为 final 的话，则无法对其进行子类化的扩展。

Spring AOP 在无法采用动态代理机制进行 AOP 功能扩展的时候，会使用 CGLIB 库的动态字节码增强支持来实现 AOP 的功能扩展。

2.2 AOP 术语（用到的时候看看就行…）

2.2.1 JoinPoint 连接点

系统运行前，AOP 的功能模块要织入到 OOP 的功能模块中，我们要知道系统在哪些执行点上执行织入操作。这些执行点就叫 JoinPoint。

JoinPoint 类型：

  • 方法调用（Method Call）：方法被调用的时候所处的程序执行点。
  • 方法调用执行（Method Call execution）：某个方法内部执行开始时点。

2.2.2 Pointcut 切点

就是告诉你要在哪些 JoinPoint 连接点织入

Pointcut 的表述方式：

  • 直接指定 JoinPont 所在方法名称：表述简单、功能单一，只限于支持方法级别 JoinPoint 的 AOP 框架，或者只是方法调用类型的 Joinpoint，或者只是方法执行类型的 JoinPoint。织入点可能多，一个一个指定不方便。
  • 正则表达式：普遍使用，如 Jboss AOP、Spring AOP等。
  • 使用特定 PointCut 表述语言：强大灵活，如 AspectJ。

2.2.3 Advice 增强/通知

理解成横切逻辑做了什么

1）Before Advice

在 JoinPoint 指定位置之前执行的 Advice 类型。通产不中断流程（必要时可以抛出异常）。如果 Before Advice 被织入到方法执行类型的 JoinPoint，其优于方法执行。

应用：系统初始化、封装安全检查的逻辑。

2）After Advice
  • After returing Advice：JoinPoint 处流程正常完成后才会执行
  • After throwing Advice：JoinPoint 处流程抛出异常后才会执行
  • After (finally) Advice：无论正常返回还是抛出异常都会执行
3）Around Advice

对 JoinPoint 进行“包裹”，在 JoinPoint 前后都指定相应逻辑，甚至于中断或忽略 JoinPoint 处原来逻辑的进行。

==2.3 Spring AOP 实现机制（重要）==

Spring AOP 属于动态 AOP，采用动态代理和字节码生成技术实现。与最初的 AspectJ 采用编译器将横切逻辑织入目标对象不同，动态代理和字节码生成都是在运行期间为目标对象生成一个代理对象，而将横切逻辑织入这个代理对象中，系统最终使用的是织入了横切逻辑的代理对象，而不是真正的目标对象。

2.3.1 设计模式——代理模式

代理处于访问者与被访问者之间，隔离两者间的直接交互，访问者与代理打交道好像在和真正的目标对象打交道一样。代理可以减轻被访问者的压力，也可以添加自己的逻辑在转发访问请求之前或之后。

假如你要拦截所有凌晨时间的request() 请求，那你在代理的增强逻辑里面加上这个时间判断就可以了。但是，可能有很多接口的实现类都有这个方法，你总不能为每种类型的接口都创建一个代理并添加相同的增强逻辑吧… …

JoinPoint 切入点相同，对应的目标对象类型不同，静态代理不建议！

Spring中的dao，service，controller的区别或联系

  • dao（数据访问层）：负责与数据打交道。通过Spring配置文件与数据库进行连接，封装了对数据库进行增删改查的基本操作。

  • service（服务层）：建立在dao层上面，controller下面，提供服务。对一个dao或多个dao进行再次封装，封装成一个服务类。

  • controller（业务层）：负责各个模块的管理控制；负责url映射地址。管理用户操作，根据用户请求来调用对应服务以便完成请求处理等；对service进行调用。

其中dao与service层在开发中，需要先进行接口的设计，接着再设计实现接口的类。

之所以这样会是因为会更符合 MVC 模式的理念，也符合大多数人的编程习惯，并且极大减少了开发的工作量，使其后期维护也较为轻松，更为重要的一点是可以极大地保障数据的安全，防止 controller 层对数据库直接进行操作。

讲讲 Spring 中的常见注解

Spring 框架中有许多注解，它们对于开发者来说都非常重要，但以下是其中一些最为常用和重要的注解：

  1. @ComponentScan：该注解用于自动扫描指定包及其子包中的组件，Spring 会将这些组件注册到容器中。通常与其他注解配合使用，如 @Component、@Service、@Repository 和 @Controller 等。
  2. @Autowired：该注解用于自动装配 Spring Beans。当 Spring 容器中存在与被装配属性类型匹配的 Bean 时，会将其自动注入到属性中。
  3. @Controller、**@Service、@Repository**：这些注解用于标识组件的角色，分别表示控制器、服务和仓库（或 DAO）。它们是 @Component 注解的特殊化，用于更清晰地指定组件的用途。
  4. @RequestMapping：该注解用于将 HTTP 请求映射到特定的处理方法上。它可以用于类级别和方法级别，用于指定处理请求的路径、HTTP 方法等。
  5. @RestController：这是一个组合注解，结合了 @Controller 和 @ResponseBody。它表示一个控制器，其所有处理方法都会将返回值直接作为 HTTP 响应的主体内容返回，而不是视图名称。
  6. @RequestParam、**@PathVariable：这些注解用于从请求中获取参数值。@RequestParam** 用于获取请求参数，而 @PathVariable 用于获取路径变量。
  7. @ResponseBody、**@RequestBody：这些注解用于在控制器中处理 HTTP 请求和响应的主体内容。@ResponseBody** 表示方法的返回值直接作为响应体，而 @RequestBody 表示将请求体内容映射到方法参数上。
  8. @Transactional：该注解用于标识事务管理的方法或类。当方法或类被标记为 @Transactional 时，Spring 会在方法调用时开启一个事务，并在方法执行结束后根据情况提交或回滚事务。
  9. @Configuration、**@Bean：这些注解用于定义 Spring 的配置类和 Bean 的定义。@Configuration** 用于标识配置类，而 @Bean 用于在配置类中定义 Bean。
  10. @Value：该注解用于从属性文件、环境变量等地方注入值到 Spring Bean 的属性中。可以用于注入基本类型、String、List、Map 等类型的值。

以上是一些在 Spring 中使用频率较高的注解，它们能够帮助开发者更便捷地编写和管理 Spring 应用程序。另外，你还可以谈谈项目中用了哪些自定义注解。

怎样解决 Spring 的循环依赖

来源：https://mp.weixin.qq.com/s/-o8N7sZ9t_EQOLLoDMOGFg

==问：什么是循环引用？==

A 依赖 B，B 依赖 A，或者 C 依赖 C，就成了循环依赖。

当然了，循环依赖只发生在 Singleton 作用域的 Bean 之间，因为如果是 Prototype 作用域的 Bean，Spring 会直接抛出异常。原因很简单，AB 循环依赖，A 实例化的时候，发现依赖 B，创建 B 实例，创建 B 的时候发现需要 A，创建 A1 实例……无限套娃。。。。

好，接下来解决循环依赖的问题。

开发人员做好设计，别让 Bean 循环依赖，但面试官既然这么傻逼地问了这个问题，肯定不想听这个最正确的答案（😂）。只能硬着头皮作答了。

==问：怎么解决循环应用？==

Spring 通过三级缓存解决循环依赖的问题：（通过提前暴露半成品Bean）

  1. 一级缓存 singletonObjects：存储已经完全初始化好的 Bean。
  2. 二级缓存 earlySingletonObjects：存储已经被实例化但还没有被完全初始化的 Bean。（属性没注入，半成品bean）
  3. **三级缓存 singletonFactorys**：存储 Bean 工厂对象，用于解决 Bean 的早期引用问题，通过 Bean 工厂可以创建出 Bean 的代理对象。当检测到循环依赖时，可以通过这个工厂得到 Bean 的代理对象，这个代理对象被放入二级缓存中。

注： 对象的 实例化：堆内存开辟了对象的空间； 初始化：属性注入完成

场景一：Bean A 和 B 循环依赖

  1. 把 A 的对象工厂放入三级缓存，表示 A 开始实例化；**singletonFactories.put("A", factory).**
  2. 发现 A 依赖 B，继而转向实例化 B，发现依赖 A，就从一级缓存里找 A，但 A 还未完全初始化好，因此 B 转而从二级缓存中获取 A，如果二级缓存也没有，则从三级缓存中获取 A 的工厂对象，通过这个工厂对象获取 A 的早期引用（可能是 A 的代理对象），并将这个早期引用放入二级缓存，同时删除三级缓存中的 A。
  3. 使用 A 的早期引用完成 B 的初始化，将 B 放入一级缓存。
  4. 继续先前的实例化 A， A 直接引用一级缓存中的 B ，将 A 放入一级缓存。

注意：B的工厂没在三级缓存出现过（为什么？？还没懂）

==问：三级缓存分别有什么作用？二级缓存不行吗？==

一级缓存：最终存放完全初始化好的单例对象，用于快速获取。

二级缓存：提前暴露未完成初始化的 Bean，防止简单的循环依赖问题。

三级缓存：提供延迟加载的能力，能够动态生成 Bean 的早期引用（例如，AOP 代理对象），并将其存入二级缓存。

三级缓存是解决循环依赖的关键，因为它能够：

  1. 动态生成 Bean 的早期引用；
  2. 支持创建代理对象，避免直接暴露原始对象。

==问：哪些循环依赖Spring无法处理？==

Spring 能解决所有循环依赖吗？

构造器注入在循环依赖中会立即要求依赖实例化，因此无法处理循环依赖。

基于 @Autowired 的注入（通常是 Setter 注入或字段注入），Spring 能够先实例化对象并在后续注入阶段通过三级缓存解决循环依赖，避免了死锁问题。

Bean 的生命周期

看这个：Spring Bean生命周期，好像人的一生、Spring Bean 生命周期 （实例结合源码彻底讲透）

普通 Java 对象的生命周期：==实例化——>垃圾回收==

Bean 的生命周期： ==**1. 实例化 ——-> 2. 设置属性 ——> 3. 初始化 ——-> （使用） ——-> 4. 销毁 **==

Bean 实例化的时机也分为两种，BeanFactory 管理的 Bean 是在使用到 Bean的时候才会实例化，ApplicantContext 管理的Bean在容器初始化的时候就完成 Bean 实例化。

我们讲到了 Bean 容器四个阶段，会有一些容器级的方法，进行前置和后置的处理，比如 InstantiationAwareBeanPostProcessor、BeanPostProcessor接口方法。这些方法独立于Bean之外，并且会注册到Spring容器中，在Spring容器创建Bean的时候，进行一些处理。

  • 实例化：第 1 步，实例化一个 Bean 对象
  • 属性赋值：第 2 步，为 Bean 设置相关属性和依赖
  • 初始化：初始化的阶段的步骤比较多，5、6步是真正的初始化，第 3、4 步为在初始化前执行，第 7 步在初始化后执行，初始化完成之后，Bean就可以被使用了
  • 销毁：第 8~10步，第8步其实也可以算到销毁阶段，但不是真正意义上的销毁，而是先在使用前注册了销毁的相关调用接口，为了后面第9、10步真正销毁 Bean 时再执行相应的方法

中间的一些扩展过程也可以分四类：

  • 一：Aware 接口：Aware 接口的作用是让 Bean 能拿到容器的一些资源，例如 BeanNameAware 可以拿到 BeanName。
  • 二：BeanPostProcessor 接口：在 Bean 的生命周期里，会有一些后处理器，它们的作用就是进行一些前置和后置的处理。
  • 三：InitializingBean 接口：InitializingBean 和 DisposableBean 接口就是用来定义初始化方法和销毁方法的。
  • 四：init-method 方法：初始化方法。
